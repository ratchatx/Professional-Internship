Use Case Specifications

ถอดรายละเอียด Main Flow / Alternate Flow จาก README: Frontend Flow + Module breakdown

UC-01

Login & Role Resolution

Core Platform
ActorStudent, Company, Advisor, Admin
คำอธิบายLoginPage รับ credential → เรียก service ตรวจสอบ → เก็บ token + role ตามที่ README กำหนด
Pre-conditionผู้ใช้มีบัญชีและอินเทอร์เน็ตพร้อมใช้งาน
Triggerผู้ใช้เปิดเว็บและกดเข้าสู่ระบบ
Main Flow
  1. กรอก Username / Password แล้วกดเข้าสู่ระบบ
  2. Front-end เรียก API service ภายใต้ api/
  3. เมื่อสำเร็จ บันทึก token + role ลง context และ localStorage
  4. Router ส่งผู้ใช้ไป layout ที่ตรงบทบาท
Alternate Flow
  • Credential ไม่ถูกต้อง → แสดง Snackbar แจ้งเตือน
  • เซิร์ฟเวอร์ตอบ role ไม่ครบ → ปิด session และให้ login ใหม่
Post-conditionผู้ใช้มี session สมบูรณ์และพร้อมใช้เมนูเฉพาะบทบาท
UC-02

Role-based Layout & Navigation

App.jsx
Actorทุกบทบาท (หลัง Login)
คำอธิบายApp.jsx ตรวจ role ก่อน render layout / sidebar / route guard
Pre-conditionมี token + role ใน context/localStorage
Main Flow
  1. Router อ่าน role จาก context
  2. เลือก layout (Admin, Advisor, Company, Student)
  3. โหลดเมนู/คอมโพเนนต์ที่ผูกกับบทบาทนั้น
  4. ป้องกันการเข้าถึงหน้าอื่นด้วย guard
Alternate Flow
  • Role ไม่ตรงกับ route → redirect กลับ dashboard ที่ถูกต้อง
  • Token หมดอายุ → logout และกลับหน้า Login
Post-conditionผู้ใช้เห็น navigation และ widget ที่สอดคล้องกับสิทธิ์เท่านั้น
UC-03

Student Request Lifecycle

Student Module
ActorStudent
คำอธิบายนักศึกษาส่งคำร้อง, อัปโหลดไฟล์, ติดตามสถานะใน Dashboard ตาม Flow README
Pre-conditionเข้าสู่ระบบด้วยสิทธิ์ Student และกรอกข้อมูลสถานประกอบการเรียบร้อย
Main Flow
  1. เปิด Student Dashboard → เลือกเมนูส่งคำร้อง
  2. กรอกฟอร์ม + แนบไฟล์, ผ่าน client validation
  3. กด Submit → API บันทึกคำร้องสถานะ "รออนุมัติ"
  4. Dashboard แสดง progress tracker (Pending → Approved)
  5. ได้รับ Snackbar แจ้งผลทุกครั้งที่สถานะเปลี่ยน
Alternate Flow
  • Validation ไม่ผ่าน → ไฮไลต์ฟิลด์ผิดและไม่ส่งคำร้อง
  • ถูก Reject → tracker แสดงสถานะไม่อนุมัติพร้อมเหตุผล
Post-conditionคำร้องถูกจัดเก็บและสามารถติดตามสถานะย้อนหลังได้
UC-04

Advisor/Admin Request Approval

Advisor & Admin
ActorAdvisor, Admin
คำอธิบายตรวจสอบคำร้องใน Dashboard, ให้ความคิดเห็น, อนุมัติหรือปฏิเสธ
Pre-conditionมีคำร้องสถานะ Pending และผู้ใช้มีสิทธิ์ที่เหมาะสม
Main Flow
  1. เข้าสู่ Dashboard ที่รวบรวมคำร้อง
  2. ใช้ตัวกรองดูรายการที่รอการตัดสิน
  3. เปิดรายละเอียดพร้อมไฟล์แนบ
  4. เลือก Approve / Reject และระบุเหตุผล
  5. ระบบอัปเดตสถานะและแจ้ง Student ผ่าน Snackbar/Notification
Alternate Flow
  • พบข้อมูลไม่ครบ → ส่งกลับ (Request changes) ให้ Student แก้ไข
  • การอนุมัติล้มเหลว → ระบบ rollback และแจ้งเตือนความผิดพลาด
Post-conditionสถานะคำร้องเปลี่ยนและถูกบันทึกในระบบ รายการถัดไปสามารถถูกประมวลผล
UC-05

Company Daily Check-in Cycle

Company Module
ActorStudent, Company
คำอธิบายStudent บันทึก check-in รายวัน / สัปดาห์ และ Company ยืนยันผ่าน Intern List
Pre-conditionStudent ถูกจับคู่กับสถานประกอบการในระบบ
Main Flow
  1. Student กรอกฟอร์ม check-in ในเมนู Student → Check-ins
  2. ระบบบันทึกข้อมูลและ sync กับ company dashboard
  3. Company เข้าหน้า Intern List
  4. เลือกนักศึกษาที่เกี่ยวข้องและกดยืนยัน (หรือ comment)
  5. Admin Dashboard ได้รับ data เพื่อ oversight
Alternate Flow
  • Company ไม่ยืนยัน → สถานะค้าง และแจ้งเตือนถึง Student
  • Student ลืมกรอก → ระบบเตือนผ่าน UI ก่อนสิ้นวัน
Post-conditionข้อมูลเวลาและการยืนยันถูกเก็บในฐานข้อมูลสำหรับการประเมิน
UC-06

Admin Oversight & Export

Admin Module
ActorAdmin
คำอธิบายAdmin ดูสถิติ, ตรวจผู้ใช้, ติดตามคำร้อง/เช็คชื่อ/การชำระเงิน และส่งออกไฟล์ XLSX
Pre-conditionมีสิทธิ์ Admin และมีข้อมูลในระบบ
Main Flow
  1. เปิด Dashboard Admin เพื่อดูการ์ดสรุป
  2. จัดการผู้ใช้/สิทธิ์ หรือปรับสถานะคำร้อง/ชำระเงิน
  3. ตรวจ log เช็คชื่อที่ sync จากบริษัท
  4. กดปุ่ม Export XLSX (helper ใน utils/)
  5. ดาวน์โหลดไฟล์และแชร์ต่อให้ส่วนงานต่าง ๆ
Alternate Flow
  • Export ล้มเหลว → Snackbar แจ้งเตือนและให้ลองอีกครั้ง
  • พบข้อมูลผิด → ใช้ action manual เพื่อแก้ไขก่อน export
Post-conditionข้อมูลได้รับการตรวจสอบ/ส่งออก พร้อมสำหรับ auditing หรือรายงาน
